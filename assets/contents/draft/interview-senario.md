## 1. 전체 프로젝트 기반 “핵심 역량 검증” 질문

#### ❖ **당신의 포트폴리오 전체를 봤을 때 반드시 물어보는 질문**

1. 여러 프로젝트에서 공통적으로 나타나는 ‘**분산 아키텍처 설계**’ 역량은 어떻게 학습하고 검증했는가?

<form class="inline-answer">
  <input type="text" placeholder="내 답 입력…" />
</form>
<details class="answer-box">
<summary>답변 예시</summary>
<div markdown="1">

> 저는 분산 아키텍처를 필요성 → 학습 → 실전 적용 → 성능 검증의 흐름으로 체계적으로 익혔습니다.

필요성 경험
단일 서버 구조에서 트래픽 증가 시 API 응답 지연·DB 병목이 반복되며 “분산해야 한다”는 문제 의식이 생겼습니다.

학습 방식
MSA, 이벤트 기반 아키텍처, CQRS, 캐시/비동기/스트리밍 구조를
시스템 설계 관점에서 공부했습니다.
특히 Redis Streams, Kafka, pgvector 등 기술 특성에 따른 트레이드오프를 정리했습니다.

실전 적용
FLIK에서는 App/Batch 서버 분리, 벡터 DB 분리, 캐시 계층 도입 등
워크로드 기반 분산 구조를 설계했습니다.
도르멍드르멍 해커톤에서는 API 서버, 음성 생성 파이프라인, 저장소를
독립 서비스 형태로 분리해 배포했습니다.

검증(Performance Validation)
JMeter, APM, Slow Query Log를 사용해
분산 전/후의 성능을 직접 측정했습니다.
예: 벡터 검색은 DB 단독 처리 대비 캐싱 후 평균 응답 시간이 40% 개선되었습니다.

이렇게 단순히 따라한 것이 아니라 “문제 해결 수단으로서” 분산을 설계하고 검증했습니다.

2. 당신이 말하는 “다차원적 성장”이란 기술적으로 무엇을 의미하는가?

<form class="inline-answer">
  <input type="text" placeholder="내 답 입력…" />
</form>
<details class="answer-box">
<summary>답변 예시</summary>
<div markdown="1">

> 제가 말하는 다차원적 성장은 **“한 영역을 깊게 이해하면서도, 인접 영역과 연결하여 전체 시스템을 최적화하는 능력”**을 의미합니다.

구체적으로는 다음 4가지입니다:

백엔드 기본기 – API, DB 설계, 트랜잭션, 성능 최적화

데이터/AI 역량 – Embedding, 추천 알고리즘, 벡터 검색

인프라/DevOps – Docker, 배포 자동화, 모니터링, Blue/Green, K8s

시스템 설계 – 캐시 계층, 워크로드 분리, 스트림 처리, 분산 구조

즉, 단순히 기술 스택이 많은 것이 아니라
서버 → DB → 캐시 → 인프라 → ML 모델이
어떻게 서로 연결되어 동작하는지 ‘전체 구조’를 이해하고 개선하는 성장입니다.

</div>
</details>

5. 기술적 선택(Spring / Redis Streams / pgvector / RDS / Raspberry Pi 등)을 할 때 어떤 기준으로 결정하는가?


<form class="inline-answer">
  <input type="text" placeholder="내 답 입력…" />
</form>
<details class="answer-box">
<summary>답변 예시</summary>
<div markdown="1">

> 저는 기술 선택을 5단계 의사결정 기준으로 진행합니다.

1) 문제 정의

정합성이 중요한가?

지연 시간이 중요한가?

확장성이 중요한가?

비용 제약이 있는가?

2) 요구사항 정량화

예: “응답 200ms 이하”, “초당 1k 요청 처리”, “데이터 정확도 95% 이상”

3) 기술 비교(Trade-off 평가)

Redis Streams vs Kafka → 순서 보장 vs 확장성

pgvector(HNSW) vs IVF vs PQ → 속도 vs 정확도

RDS vs 자체 DB → 운영 책임 분산

Spring vs FastAPI → 개발 생산성 vs 실행 속도

4) 운영/모니터링 난이도

모니터링 가능 여부

장애 복구(Process 친화도)

슬랙/문서 커뮤니티의 강도

5) 테스트 후 지표 기반 최종 결정

P90/P99 latency

CPU/IO 비율

캐시 hit ratio

검색 정확도(precision@k)

기술 선택에 감을 배제하고
**“문제 → 요구사항 → 비교 → 지표 검증”**의 구조로 결정합니다.
